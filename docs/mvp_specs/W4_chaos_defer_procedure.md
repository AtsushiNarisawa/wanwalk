# §7.2 残 DEFER 4 件 実機 E2E 手順書

A3 仕様書 §7.2 「エラー注入テスト」12 項目のうち、Simulator 単独実施が困難な 4 件（#1 / #5 / #9 / #10）を CEO 実機で W4 にまとめて検証する手順書。

## 前提

| 項目 | 値 |
|---|---|
| 対象 Build | TestFlight 1.1.0 (33) もしくは W4 提出予定の最新 Build |
| 対象端末 | CEO 手持ち実機 A or B（§7.3.1 参照） |
| iOS バージョン | CEO 機材の実 OS（W4 着手時に CTO に共有） |
| Sentry プロジェクト | `wanwalk / flutter` |
| アカウント | 普段使い or `test@dog-hub.shop`（パスワード `WanWalkTest!2026`） |
| ネット接続 | 通常時 Wi-Fi or モバイル回線（項目 1 のみ意図的に切断） |
| 所要時間 | 全 4 件で約 45-60 分 + 項目 9 のみ別途 30 分待機 |

## 共通の判定基準

| 判定 | 条件 |
|---|---|
| 🟢 PASS | 期待動作通り・**クラッシュなし**・エラー画面 or 自動復帰・UX 上違和感許容範囲内 |
| 🟡 WARN | クラッシュなしだが UX 改善要（文言・遷移先など）・公開後改善で OK |
| 🔴 FAIL | クラッシュ発生 or アプリ完全ハング or データ消失 → ホットフィックス対象 |

## 共通のエビデンス保存ルール

各項目につき以下を残す:
1. 実行前後のスクリーンショット最低 2 枚（「注入前」「注入後の画面」）
2. Sentry に event が記録された場合は event URL or event ID
3. 完了時刻と所要分秒
4. メモ（気になった挙動・想定外の遷移など）

保存先: `wanwalk-app/docs/mvp_specs/W4_chaos_defer_evidence/`
ファイル命名: `chaos_NN_<short_label>_<seq>.png`（例: `chaos_01_network_offline_01.png`）

実施後 CTO スレッドで本ファイルに「結果サマリー」セクションを追記して commit する。

## #1 ネットワーク切断 → ルート詳細

### 注入内容
ルート詳細画面を開く直前にネット接続を切る → どう振る舞うか確認。

### セットアップ
1. アプリ起動 → ホームタブ → 公式ルート一覧を一度スクロールして表示（キャッシュ作成）
2. 設定アプリ → Wi-Fi OFF + モバイルデータ通信 OFF（**機内モード ON が手っ取り早い**）
3. アプリに戻る

### 実行
1. ホームのフィードカード or 「公式ルート」タブから **未キャッシュのルート**（一覧で 1 回も詳細を開いていないルート）をタップ
2. 画面遷移後の挙動を観察
3. **追加チェック**: そのまま戻る → 別の未キャッシュルートをもう 1 件開く（再現性確認）
4. **追加チェック**: 「散歩記録」「ピン投稿」など他の画面にも遷移して、ネットワーク必要な操作が落ちないか確認

### 期待動作
- 🟢 ローディング → エラー画面（A5 文言「申し訳ありません 通信エラー」系）or キャッシュ表示
- 🟢 再試行 / 戻る ボタンが表示される
- 🟢 クラッシュなし・他画面に遷移してもアプリは生きている
- 🔴 ホワイトアウト・無限ローディング・強制終了 → FAIL

### 後始末
- 機内モード OFF → Wi-Fi or モバイルデータ復帰
- ルート詳細を再度開いて正常表示することを確認

### エビデンス
- `chaos_01_network_offline_before.png`（機内モード ON 直前のホーム）
- `chaos_01_network_offline_error.png`（エラー画面 or 表示状態）
- `chaos_01_network_offline_recovery.png`（ネット復帰後の正常表示）

## #5 カメラ権限拒否

### 注入内容
カメラ権限を OFF にしたうえで「ピン投稿」「写真追加」系のフローを実行 → 拒否時の説明モーダルと設定誘導が機能するか。

### セットアップ
1. iOS 設定アプリ → **WanWalk** → カメラ → **OFF**（既に許可済みなら OFF にする）
2. 設定 → 写真 はそのまま（カメラだけ拒否状態にする）
3. アプリ起動

### 実行
1. 散歩記録中 or 散歩完了後の写真追加導線、もしくは「ピン投稿」フローでカメラを起動する操作を実行
2. 表示されるダイアログ・モーダルを確認
3. 「設定を開く」「OK」「キャンセル」ボタンを順に試す
4. 「設定を開く」が iOS 設定アプリの WanWalk 画面に飛べるか確認

### 期待動作
- 🟢 「カメラへのアクセスが必要です」系の説明モーダル表示
- 🟢 「設定を開く」ボタンで iOS 設定の WanWalk 画面に直接遷移
- 🟢 「キャンセル」or 戻るで前画面に戻る・クラッシュなし
- 🟡 説明文言が短すぎる/長すぎる場合は WARN（公開後文言改善で OK）
- 🔴 ダイアログ出ずに白画面・カメラ画面のままハング・強制終了 → FAIL

### 後始末
- 設定 → WanWalk → カメラ → **ON** に戻す
- 通常のピン投稿フローでカメラが起動できることを確認

### エビデンス
- `chaos_05_camera_denied_settings.png`（iOS 設定で カメラ OFF 状態）
- `chaos_05_camera_denied_modal.png`（拒否説明モーダル）
- `chaos_05_camera_denied_redirect.png`（「設定を開く」遷移後の iOS 設定画面）

## #9 アプリをバックグラウンドに 30 分放置 → 復帰

### 注入内容
長時間バックグラウンドに置いた後の復帰挙動。Supabase JWT 再認証 / Realtime 再接続 / GPS セッション復旧 などが落ちずに走るか。

### セットアップ
1. アプリ起動 → ログイン状態でホームタブを開く
2. **ストップウォッチを開始**（30 分カウント用）
3. ホームボタン（or スワイプアップ）でアプリをバックグラウンドへ
4. iPhone を画面ロック状態で放置（充電中推奨）

### 実行
1. 30 分経過後（28-32 分の範囲で OK）に iPhone のロック解除
2. WanWalk アプリを TAP で復帰（タスクスイッチャーから戻すのではなくアイコン TAP）
3. 復帰後の挙動を 2-3 分観察
4. **追加チェック**: ホーム → ルート詳細 → ピン投稿の導線まで一通り操作してみる
5. **追加チェック**: 再ログインを求められた場合は、求められた事実をメモ（PASS 条件には影響しない）

### 期待動作
- 🟢 ホーム画面が即座に表示される（白画面 → コンテンツの遷移は許容）
- 🟢 ルートデータ・ユーザー情報が再ロードされる
- 🟢 セッション期限切れの場合は自動再認証 or ログイン画面誘導
- 🟢 クラッシュなし・データ消失なし
- 🟡 ローディング 5-10 秒以上は WARN（公開後 UX 改善で OK）
- 🔴 ホワイトアウト・強制終了・操作不能 → FAIL

### エビデンス
- `chaos_09_background_before.png`（バックグラウンド移行直前のホーム）
- `chaos_09_background_resume.png`（30 分後の復帰直後のホーム）
- `chaos_09_background_navigation.png`（復帰後にルート詳細まで遷移できた状態）

### 補足
- もし Supabase のセッション再認証が走った場合、`auth_provider.dart` の AuthChangeEvent listener（day 13 §7.2 #2 で動作確認済）が機能していることの実機での裏取りにもなる
- 30 分待ちの間に他項目（#10 など）を並行実行しても良い

## #10 容量不足（写真保存先）

### 注入内容
iPhone のストレージを意図的に圧迫し、写真撮影 → 保存のフローでどう振る舞うか確認。

### セットアップ（**実機ストレージを潰すため最後に実施推奨**）
1. iOS 設定 → 一般 → iPhone ストレージ で「使用可能」を確認（W4 着手前メモ）
2. 写真アプリ or ファイルアプリ等で大量ダミーデータを作る、もしくは:
   - **より安全な代替手段**: 開発者向け iOS シミュレータ的な方法は実機では難しいので、**実機ストレージが既に逼迫している状況（残り 1GB 未満）で実施**するか、**写真アプリで「動画を 10-20 本撮影」してギャラリーを膨らませる**等で代替
3. **完全に容量 0 にする必要はない**: 残り 50-100MB まで圧迫すれば、Supabase Storage アップロード or 端末側一時保存で容量不足エラーが再現する可能性が高い

### 実行
1. アプリで「ピン投稿」or「散歩記録の写真追加」フローを起動
2. カメラ起動 → 撮影 → 保存ボタン押下
3. 表示されるエラー or 挙動を観察

### 期待動作
- 🟢 「保存に失敗しました」「容量が不足しています」系のエラー画面 or トースト表示
- 🟢 投稿フローからキャンセル可能・前画面に戻れる
- 🟢 クラッシュなし
- 🟡 エラー文言が技術的すぎる場合は WARN（A5 文言「申し訳ありません」基調に合わせて公開後改善で OK）
- 🔴 ハング・強制終了・ピン投稿データ消失（書きかけが完全に飛ぶ）→ FAIL

### 後始末
- 撮影したダミー動画 / 写真を削除してストレージを復旧
- iOS 設定 → 一般 → iPhone ストレージ で復旧確認

### エビデンス
- `chaos_10_storage_full_before.png`（iOS 設定のストレージ画面・残量表示）
- `chaos_10_storage_full_error.png`（容量不足エラー画面 or トースト）
- `chaos_10_storage_full_recovered.png`（ストレージ復旧後の正常保存）

### 補足
- **#10 は実施コストが高い**（実機ストレージを潰すため日常業務に影響）。CEO のスケジュールと相談し、本項目だけ **公開後 Sentry テレメトリで代替検出**でも OK と判断する場合は スキップ → §7.3.4 の事後検知運用に委ねる
- スキップ判定の場合、本ファイルの「結果サマリー」で `#10: SKIP（公開後 Sentry 運用へ委譲）` と記録

## FAIL 発生時の hotfix 判断フロー

1. FAIL を確認 → 当該項目の Sentry event があれば URL を控える
2. CTO スレッドで「ホットフィックス着手判断」を CEO に確認
3. 着手判断後:
   - 原因特定（Sentry stack trace + 該当コード grep）
   - 修正 + flutter analyze
   - 該当項目の再検証（Sim or 実機・本書手順を再実行）
   - Build N+1 提出フロー（[skills/flutter-build.md](../../../.claude/skills/flutter-build.md) §3 + Sentry section）
4. WARN は MVP 公開ブロッカーではないので公開後の改善キューに入れる
